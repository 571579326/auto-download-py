# commit-rules skill

## PR提交原则

所有代码改动必须**分功能、分批**提交，禁止一次性提交多个不相关的功能。每个 PR 只包含一个独立功能或修复的变更。

## PR提交标题格式

```
[操作]【模块名称(中文)】PR说明
```

- **操作**：优化 / 新增 / 修改 / 删除 / 重构 / 修复 / 升级
- **模块名称(中文)**：对应当前改动的功能模块，如 业务图像点击服务、项目文档、测试与脚本、浏览器管理、业务流程 等
- **PR说明**：用中文简要描述本次提交的核心变更，如"将图像点击逻辑从BusinessService抽离为公共业务服务层"

### 示例

```
[重构]【业务图像点击服务】将图像点击逻辑从BusinessService抽离为公共业务服务层
[新增]【公共服务能力文档】新增business-image-click能力文档，同步更新项目文档引用
[优化]【项目文档】更新README、架构文档和Skill文档，同步项目版本与依赖锁
[新增]【测试与脚本】新增业务流程单元测试、E2E测试脚本和数据库检查工具
[新增]【浏览器管理】新增纯净模式、Selenium短接管，扩展配置项并注册业务路由
[修复]【页面定位】修复pageId为空时默认取active页的逻辑错误
```

## PR提交信息规范

### 首行

首行即 PR 标题，**不超过 100 字符**。

### 正文

正文空一行后列出主要变更点，每条用 `- ` 开头，不超过 80 字符每行。

```
[重构]【业务图像点击服务】将图像点击逻辑从BusinessService抽离为公共业务服务层

- 新增 business_image_click_service.py：封装公共图像点击服务
- 重构 business_service.py：移除内联代码，复用新服务层
- 新增 imageClickSkipped/imageClickSkipReason 响应字段
```

### 标注新文件

新增文件在正文后附加说明 `create mode 100644 <file_path>`。

## 分功能分批规则

1. **一个 PR 一个功能**：代码变化如涉及多个模块功能，拆分为多个独立 PR
2. **代码与文档分离**：代码功能改动和对应的文档更新可以合并为一个 PR，但文档独立优化应单独提交
3. **频繁提交小更改**：不要等积累大量改动再提交
4. **提交粒度参考**：
   - 每次新增或重构一个 service 方法 -> 一个 PR
   - 每次新增一个 API 路由 -> 一个 PR（含 schema、manager、service、API）
   - 文档批量调整 -> 一个 PR

## 注意

- 始终先 `git status` 确认工作区干净再开始新功能开发
- 提交前确认不包含调试代码（print、pdb、TODO 标记等）
- 排除无关文件（如 .zip、.log、__pycache__）
